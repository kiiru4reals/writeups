### Branching strategies in a nutshell
There are many branching strategies but all of them boil down to the following effects:
1. Optimize for individual productivity: This is where a branch holds a feature and is merged to the main branch when everything is ready to be deployed. This makes it possible for independent features to work but if a feature takes long to be developed there would be conflicts.
2. Optimize for team productivity: This is where you have one branch that everyone works on and pushes everything to it. This is good because the main branch is always up to date but if things go wrong, the entire branch has an issue, so development needs to stop so that the problem can be resolved.

It is recommended that you break down functionality into small bits that can be merged to the main branch multiple times in a day. You can have a configuration that rejects merging commits that may introduce breaking changes.
